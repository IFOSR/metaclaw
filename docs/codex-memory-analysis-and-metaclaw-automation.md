> 聚焦：Codex 的记忆自动化机制，MetaClaw 如何降低人工介入成本  
> 明确不学：Project 级记忆隔离、云端同步

## 1. Codex 记忆机制全景

### 1.1 记忆生命周期

```
用户与 Codex 交互
       ↓
Codex 自动检测「值得记住的信息」
  （偏好、项目约定、常用命令、代码风格等）
       ↓
通过 function calling 写入记忆（自然语言条目）
       ↓
存储在 OpenAI 服务端
       ↓
后续会话中，通过语义检索召回相关记忆
       ↓
自动注入 System Prompt，用户无感知
       ↓
用户可通过 /memory 命令管理（非必经流程）
```

### 1.2 核心特征

- **自动提取**：无需用户手动触发或确认，LLM 在对话中自行判断什么值得记
- **自动召回**：相关记忆静默注入上下文，不弹确认框
- **自然语言存储**：不是结构化字段，而是自然语言句子（如 "用户偏好 TypeScript strict mode，不喜欢 any 类型"）
- **LLM 驱动**：记忆的写入和召回都靠 LLM 自身的判断能力，不依赖规则引擎
- **极简交互**：用户感知层面是「Codex 好像记得我的偏好」，而非「我需要管理一个记忆系统」

## 2. Codex 自动化的核心——「用户零负担」

### 2.1 Codex 做了什么，用户不需要做

| 环节 | Codex 做法 | 用户感知 |
|------|-----------|---------|
| **记忆写入** | LLM 自动判断 + function calling 写入 | 零感知 |
| **记忆召回** | 服务端语义检索，自动注入 System Prompt | 零感知 |
| **记忆更新** | 新信息覆盖旧信息，自动去重 | 零感知 |
| **记忆过期** | LLM 判断旧记忆是否被新指令覆盖 | 零感知 |
| **唯一的管理入口** | `/memory` 命令（可选） | 只在需要时主动查看 |

### 2.2 关键设计原则

1. **默认静默**。除非用户主动问「你记得什么」，否则记忆操作全程静默。
2. **LLM 是法官**。不是规则引擎、不是关键词匹配、不是阈值投票。记忆的写入、召回、更新全部由 LLM 自身的语义理解来驱动。
3. **低风险全自动，高风险才需要信号**。普通偏好（代码风格、输出格式）直接 auto-apply；涉及支付、删除、外发的记忆才会触发安全策略。
4. **自然语言 > 结构化字段**。一条记忆就是一句话，不需要 scope/subject/type/confidence 等元数据。这让 LLM 理解记忆和延伸记忆都很自然。

## 3. MetaClaw 当前的交互成本分析

### 3.1 用户必须参与的节点

```
用户输入任务
   ↓
【节点 1】操作提案确认（y/n/r）
   ↓
【节点 2】记忆召回确认（y/n/s/a/r）
   ↓
任务执行
   ↓
【节点 3】偏好候选确认（模式检测达到 3 次后弹确认）
   ↓
【节点 4】学习候选审核（/learning approve/reject）
```

**每个节点都是一次打断。** 在连续工作中，这些打断的累积成本很高——尤其节点 2 和节点 3，是用户抱怨最多的「过度打扰」来源。

### 3.2 成本来源归类

| 成本来源 | 具体表现 | 频率 |
|---------|---------|------|
| 召回确认过度触发 | 关键词命中就弹确认，无关记忆也弹 | 高频 |
| 偏好确认等待 3 次 | 明明已经形成模式，还要等到第 3 次才弹 | 中频 |
| 学习候选需人工审核 | Skill patch 要用户 approve/reject | 低频但摩擦大 |
| 操作提案 | 每次启动/任务完成都弹建议 | 中频 |

## 4. 借鉴方向：把 MetaClaw 推向「静默自动化」

### 4.1 核心思路

```
当前：MetaClaw 的所有记忆决策都需要用户确认
目标：MetaClaw 只在不确定或高风险时请求确认，其余全部静默
```

这与 MetaClaw 优化方案中「三态决策」的主线一完全吻合。Codex 的实践验证了这条路是对的。

### 4.2 具体借鉴点

#### 借鉴 1：LLM 驱动记忆写入，去掉 3 次阈值

**Codex 做法**：LLM 在一次对话中就能判断「这条信息值得记住」，不需要等重复 3 次。

**MetaClaw 当前**：必须等同一个 pattern 出现 3 次才弹确认（`CONFIRM_THRESHOLD = 3`）。

**建议改造**：在 `MemoryEngine.observe()` 中引入 LLM 快速通道。当 LLM judge 可用时：

```
检测到 pattern
   ↓
LLM 判断：「这是一条明显的长期偏好吗？」
   ↓
是 → 直接生成 PreferenceApplicability，标记 confidence >= 0.85
   ↓
走 auto_apply 或低优先级 ask_review，不需要凑 3 次
```

3 次阈值只作为 LLM 不可用时的 fallback，不再作为主路径。

#### 借鉴 2：召回侧默认 auto_apply，而非默认 ask_review

**Codex 做法**：记忆自动注入，用户不需要确认。

**MetaClaw 当前**：召回成功 → 弹 recall review → 等用户 y/n。

**建议改造**（与三态决策方案一致，此处强调优先级）：

```
候选召回成功
   ↓
先走 auto_apply 判断（而非先走 ask_review）
   ↓
只有 auto_apply 条件不满足时，才降级到 ask_review
   ↓
完全无关的走 suppress
```

**心态从「默认要确认」变为「默认自动采用，不放心才确认」。**

#### 借鉴 3：静默日志替代弹窗确认

**Codex 做法**：记忆操作完全静默，用户想了解时主动 `/memory`。

**MetaClaw 建议**：`auto_apply` 的记忆不弹窗，但在任务详情中展示一个可折叠的「已自动采用的记忆」列表。用户事后审计，不是事前打断。

```
当前任务详情
─────────────────
任务: 整理 MetaClaw 优化方案
...
📋 已自动采用记忆 (3)
  ├─ [pref_xxx] 方案先给结论再列细节  ← auto_apply, score 0.91
  ├─ [pref_yyy] MetaClaw 文档使用中文  ← auto_apply, score 0.88
  └─ [pref_zzz] 保留风险栏目和经营数据 ← auto_apply, score 0.85
```

#### 借鉴 4：自然语言记忆存储

**Codex 做法**：记忆是自然语言句子，不是结构化字段。

**MetaClaw 当前**：Preference 有 `scope/subject/type/content/confidence/appliesToIntents/appliesToDomains` 等大量字段。

**建议**：在 Markdown Vault（优化方案主线三）中，Frontmatter 保留关键元数据，正文以自然语言描述记忆。同时在召回侧增加一条规则：**自然语言相似度 > 字段匹配度**。让 LLM 直接理解记忆内容，而不是通过 scope/projectId 等字段去筛选。

## 5. MetaClaw 自动化路线图（与优化方案对齐）

```
P0  记忆召回降噪（三态决策）
    → 关键改动：默认 auto_apply 而非 ask_review
    → 心态转变：静默是常态，确认是例外

P1  Evidence 基础
    → 每次自动采用必须留下 evidence
    → 让静默操作可审计

P2  LLM 驱动记忆写入
    → 去掉 3 次阈值主路径
    → LLM judge 可用时，一次即可写入

P3  静默日志替代弹窗
    → auto_apply 记忆展示在任务详情中
    → 不弹窗、不打断、但可追溯

P4  Markdown Vault（自然语言记忆）
    → 自然语言描述 > 结构化字段
    → 让记忆更容易被 LLM 理解和延伸
```

---

## 6. 最终形态

经过上述改造，MetaClaw 的交互模型从：

```
「每次决策都要用户确认」
```

变为：

```
「MetaClaw 静默记住偏好 → 自动采用 → 事后可审计
  只在不确定或高风险时才请求用户介入」
```

这就是 Codex 记忆哲学在 MetaClaw 上的落地——不是复制 Codex 的云端架构，而是学习它的**自动化心态**：让 LLM 承担更多的判断责任，把人工介入压缩到真正需要的临界点上。
