# Metaclaw TUI Spec V1

## 1. 目标

本文档定义 Metaclaw V1 的 TUI 交互规范，包括：
- 启动体验
- 对话交互模式
- 任务盘面
- 斜杠命令体系
- 自然语言与命令映射关系

PRD 见：`metaclaw-os_prd_v2.md`
技术设计见：`metaclaw-os_tech_design_v1.md`

---

## 2. 交互定位

Metaclaw V1 是交互式 TUI，类似 Claude Code 的命令行对话界面。

V1 面向：
- 高频被打断、愿意使用命令式界面的 AI power users
- 技术型创始人 / 开发者 / 研究者 / AI 高级用户

TUI 的核心价值不是“炫技”，而是：
- 快速创建任务
- 随时挂起和恢复任务
- 低摩擦查看任务盘面
- 在当前会话内完成主动编排

### 2.1 交互硬规则
- 输入框始终可用，不因某个任务处于 `RUNNING` 或 `BLOCKED` 而被禁用
- 任务状态是任务级概念，不是会话级锁
- 单执行器不等于单任务；系统可同时维护多个任务，只是同一时刻只运行一个
- `BLOCKED` 只阻止该任务继续执行，不阻止用户继续提交新任务

## 3. 启动体验

每次启动 Metaclaw，默认展示一屏简洁盘面摘要：

```text
┌─ Metaclaw v1.0 ─────────────────────────────────┐
│ 你有 3 个活跃任务，1 个 Blocked。                  │
│ 建议优先：#12 行业分析摘要 (进度 70%)              │
└──────────────────────────────────────────────────┘
```

随后进入交互输入区：

```text
> 
```

启动后还应展示轻量运行摘要：

```text
当前执行: 无
待执行: 0
阻塞: 0
```

### 启动时应展示的信息
- 活跃任务数
- Blocked 任务数
- 一个最高优先级建议

### 启动时不应做的事
- 冗长欢迎语
- 连续弹多个提醒
- 自动执行高风险动作

## 4. 基本交互模式

### 示例对话流

```text
$ metaclaw

┌─ Metaclaw v1.0 ─────────────────────────────────┐
│ 你有 3 个活跃任务，1 个 Blocked。                  │
│ 建议优先：#12 行业分析摘要 (进度 70%)              │
└──────────────────────────────────────────────────┘

> 帮我把这三份合同的风险点整理成对比表
  任务 #21 已创建：合同风险对比分析
  → 已进入候选队列
  → 派发给 codex-cli...

> /attach ~/docs/contract-a.pdf ~/docs/contract-b.pdf
  已关联 2 个文件到任务 #21

> 上个月的销售数据汇总是多少
  任务 #22 已创建：销售数据查询
  → 高优任务到达：用户显式插入
  → 任务 #21 已挂起 [PARKED]：被更高优任务抢占
  → 派发给 codex-cli...
  → 完成。[展示结果]
  → 恢复任务 #21
  上次进度：合同 A 已分析完成，合同 B 进行中
  中断原因：被任务 #22 抢占
  建议下一步：继续处理合同 B

> 这个先放着，等客户补资料再继续
  任务 #21 已标记为 [BLOCKED]
  → 阻塞原因：等待客户补充证据文件
  → 当前无更高优任务时，将自动切换到下一个 READY 任务
```

## 5. 主动编排触达方式

V1 的主动编排仅限：
- 启动时盘面主动
- 会话内主动
- 任务完成后的推荐
- 用户显式查询后的总结

### 支持的触达方式
1. **启动时盘面**
2. **任务完成后建议下一个任务**
3. **当前会话中的 opportunistic reminder**
4. **`/dashboard` 主动汇总**

### V1 不支持
- 后台系统通知
- 独立 daemon 驱动的主动提醒
- 跨会话推送
- 文件到达自动弹通知

## 6. 任务盘面规范

### 盘面结构
```text
┌─ Metaclaw 任务盘面 ─────────────────────────────┐
│                                                   │
│ 建议优先处理                                       │
│   #12 行业分析摘要 [READY] 进度 70%               │
│       → 资料齐全，继续成本最低                      │
│                                                   │
│ 当前卡住                                          │
│   #15 起诉书草稿 [BLOCKED]                        │
│       → 缺少：补充证据.docx                        │
│                                                   │
│ 可以稍后处理                                       │
│   #18 周报整理 [READY] 无截止时间                  │
│   #20 竞品调研 [READY] 刚创建                      │
│                                                   │
└──────────────────────────────────────────────────┘
```

### 必须呈现的信息
- 优先任务
- 当前执行中的任务
- Blocked 任务
- 推荐原因
- 状态标识

### 不应呈现的信息
- 大量底层执行日志
- 难以理解的内部评分细节
- 过多噪声任务

### 6.1 运行摘要区
任务盘面之外，主界面应始终保留一块轻量运行摘要：

```text
当前执行: #21 合同风险对比
待执行: 3
阻塞: 1
```

该区域的职责是：
- 告诉用户“系统现在在做什么”
- 不替代任务清单
- 不占用主输入框

## 7. 斜杠命令体系

## 7.1 任务管理
```text
/tasks                     -- 查看所有任务列表
/tasks active              -- 查看活跃任务
/tasks ready               -- 查看待执行候选任务
/tasks parked              -- 查看已挂起任务
/tasks blocked             -- 查看阻塞任务
/tasks done                -- 查看已完成任务
/task <id>                 -- 查看任务详情
/task <id> pause           -- 暂停任务
/task <id> resume          -- 恢复任务
/task <id> block <原因>    -- 标记任务为阻塞
/task <id> unblock         -- 解除阻塞
/task <id> cancel          -- 取消任务
/task <id> done            -- 标记完成
```

`/tasks` 默认输出应按以下分组展示，而不是平铺：

```text
任务清单：

当前执行
  #21 [RUNNING] 合同风险对比
    → 原因：最高优先级

待执行
  #30 [READY] 客户回复草稿
    → 原因：截止时间临近

已挂起
  #12 [PARKED] 行业分析摘要
    → 原因：被高优任务抢占

已阻塞
  #15 [BLOCKED] 起诉书草稿
    → 阻塞：等待补充证据.docx
```

## 7.2 偏好记忆
```text
/memory                    -- 查看所有活跃偏好（按作用域分组）
/memory search <关键词>    -- 搜索偏好
/memory add <内容>         -- 手动添加偏好
/memory edit <id>          -- 修改偏好
/memory delete <id>        -- 删除偏好
/memory candidates         -- 查看待确认偏好
/memory confirm <id>       -- 确认候选偏好
/memory reject <id>        -- 拒绝候选偏好
/memory stats              -- 偏好统计
```

## 7.3 全局命令
```text
/dashboard                 -- 任务盘面（优先级排序 + 卡点 + 建议）
/attach <文件路径>         -- 关联文件到当前任务
/history                   -- 查看交互历史
/config                    -- 查看/修改配置
/help                      -- 帮助
```

## 8. 自然语言与命令映射

自然语言是默认交互方式，斜杠命令是精确控制方式。两者等价。

### 示例映射
- “暂停” / “先放一下” → `/task <current> pause`
- “继续刚才的” → `/task <last_parked> resume`
- “现在该做什么” → `/dashboard`
- “记住：张总喜欢正式语气” → `/memory add 张总偏好正式语气`
- “看看现在有哪些任务” → `/tasks`
- “看一下排队的任务” → `/tasks ready`

### 设计原则
- 能自然语言就不强迫命令
- 命令用于提效，不用于替代自然语言
- 失败时应回显系统如何理解了用户意图

## 9. 偏好交互规范

### 候选偏好确认提示
当系统检测到重复模式并满足三次确认原则后，应给出类似提示：

```text
我注意到你已经 3 次要求“给张总写邮件时使用正式语气，并抄送法务王某某”。
要把它记为长期偏好吗？

[y] 确认  [n] 忽略  [e] 编辑后确认
```

### 注入透明性
当某次任务注入了偏好，系统应允许用户查看：
- 注入了哪些偏好
- 来自哪个作用域
- 为什么命中了这些偏好

### 9.1 恢复摘要规范
恢复任务时，系统应固定展示：
- 这是什么任务
- 上次做到哪
- 为什么停下
- 为什么现在恢复它
- 接下来准备做什么

---

## 10. 错误与边界提示

### 应提示的典型情况
- 缺少必要材料
- 当前没有可恢复任务
- 当前没有候选偏好
- 偏好冲突且系统无法可靠判断
- 当前任务不存在或已归档
- 当前任务被高优任务抢占
- 当前任务因阻塞被自动切换出去

### 错误提示原则
- 说清问题
- 给下一步操作建议
- 不暴露无意义的内部实现细节

---

## 11. 扩展性设计

命令体系采用：
`/<namespace> <action> <args>`

便于未来扩展：
- `/executor list`
- `/executor add`
- `/trigger add`
- `/export <task_id> markdown`

---

## 12. V1 交互边界总结

### 必须做到
- 自然语言创建任务
- 任务挂起 / 恢复
- `/dashboard` 盘面可用
- `/memory` 相关操作可用
- 任务完成后给出下一步建议
- 在会话内展示 Blocked / Ready 任务

### 不做
- GUI
- 后台主动推送
- 系统级通知
- 复杂多窗格布局
- 过度花哨的终端 UI

TUI 的目标是：**低摩擦、可信、能长期使用**，不是做成一套花哨的终端仪表盘。
