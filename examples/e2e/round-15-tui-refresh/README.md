# Round 15: TUI Refresh

目标：验证新的终端信息层级是否能清楚区分用户输入、系统路由、执行进度、等待状态和结果块。

## 真实验收

1. 用真实 `codex-cli` 运行：
   `./metaclaw.sh start`
2. 输入真实任务：
   `请帮我总结一下当前这个项目的定位`
3. 预期结果：
   用户输入以 `> ` 独立成块显示
4. 预期结果：
   运行摘要显示为一行数字统计：
   `当前执行 1 | 待执行 0 | 已挂起 0 | 阻塞 0`
5. 预期结果：
   输入区上方出现 `status: running codex-cli`
6. 预期结果：
   执行准备与进度使用分层前缀：
   - `→ 正在回忆任务 ...`
   - `→ 派发给 codex-cli...`
   - `· #task_xxx 已启动 codex-cli 执行器`
   - `· 正在等待执行器返回...`
7. 预期结果：
   完成后以结果块收口：
   - `✓ 任务完成 (xx.xs)`
   - `┌─ 任务结果 ─ ...`
   - 结果正文独立显示，不和执行日志混在一起

## 多任务真实验收

1. 启动真实 TUI：
   `METACLAW_HOME=/tmp/metaclaw-e2e-round15-multitask node dist/index.js`
2. 先输入一个长任务：
   `我要先调研下比亚迪与宁德时代在新能源电池层面的市场份额，并输出一个简短结论`
3. 当界面进入：
   - `当前执行 1 | ...`
   - `status: running codex-cli`
   - `· 正在等待执行器返回...`
   后，立刻继续输入第二条高优任务：
   `紧急：帮我分析佛塑科技和新能源电池产业链的关系，先给我一个一页结论`
4. 预期结果：
   第二条输入提交后，不会只停留在用户回显，必须继续出现系统反馈：
   - `→ 任务 #... 已创建`
   - `→ 正在回忆任务 #...`
   - `→ 高优任务到达，抢占当前任务 #...`
   - `→ 任务 #... 已挂起，开始执行 #...`
5. 预期结果：
   状态栏要同步反映抢占结果：
   - `当前执行 1 | 待执行 0 | 已挂起 1 | 阻塞 0`
   - `最近事件 开始执行任务 #<新任务>`

## 本轮实测记录

- 已用真实 `codex-cli` 跑通单任务场景
- 已用真实 `codex-cli` 跑通多任务高优抢占场景

  - `status: idle / running codex-cli`
  - 准备阶段 `→`

  - 静默等待提示 `· 正在等待执行器返回...`
